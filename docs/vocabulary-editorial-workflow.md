# Vocabulary source and editorial workflow

This document is the implementation contract for the vocabulary backlog. Google
Sheets is the source of truth. Discord is an ingestion channel only: accepted
messages are normalized and appended to the sheet for human review. Neither
this contract nor the sample data enables scheduling or public publishing.

## Workbook

Create a workbook with three protected tabs:

- `content_items`: one row per word sense and context. Import
  `samples/vocabulary-content-items.csv` to bootstrap the first ten rows.
- `ingestion_log`: `event_id`, `source`, `source_ref`, `payload_hash`,
  `content_id`, `result`, `created_at`. Only the ingestion automation writes it.
- `controlled_values`: allowed languages, parts of speech, statuses,
  `review_status` values, template IDs and context topics for data validation.

The `content_items` header, in order, is:

```text
content_id,word,normalized_word,language,part_of_speech,sense_key,meaning_vi,context_topic,example_en,example_vi,ipa,hook,cta,template_id,source,source_ref,license_note,publish_ok,review_status,reviewed_by,publish_at,status,claimed_at,render_id,publish_id,published_at,error_code,dedupe_key,created_at,updated_at
```

Field rules:

- `content_id`: immutable UUID generated by ingestion or the sheet helper.
- `normalized_word`: Unicode NFKC, trim, collapse internal whitespace and
  lowercase using an English locale. Preserve the display form in `word`.
- `sense_key`: stable editor-owned slug describing one dictionary sense, for
  example `financial_institution` or `river_edge`. It must not be a row number.
- `source_ref`: a direct dictionary/reference URL used to verify the sense,
  pronunciation and usage. Editors write original Vietnamese explanations and
  examples; they do not copy source example sentences.
- `license_note`: records how the source was used, for example
  `Reference only; definitions/examples paraphrased by editor`.
- `dedupe_key`: join normalized `normalized_word`, `part_of_speech`,
  `sense_key`, and `context_topic` with `|`.
- All timestamps are ISO 8601 UTC. Blank operational IDs/timestamps stay blank,
  not `null`, in Sheets.

Protect formula/automation-owned columns (`normalized_word`, `dedupe_key`,
operational timestamps/IDs and `error_code`). Apply a uniqueness check to both
`content_id` and `dedupe_key`; conditional formatting should flag duplicates.

## Discord ingestion contract

Accept one fenced `vocab` block per message:

````text
```vocab
word: bank
language: en
part_of_speech: noun
sense_key: financial_institution
meaning_vi: ngân hàng; tổ chức nhận, giữ và cho vay tiền
context_topic: personal_finance
example_en: I deposited my first paycheck at the bank.
example_vi: Tôi gửi tháng lương đầu tiên vào ngân hàng.
source_ref: https://dictionary.cambridge.org/dictionary/english/bank
```
````

Required keys are `word`, `language`, `part_of_speech`, `sense_key`,
`meaning_vi`, `context_topic`, `example_en`, `example_vi`, and `source_ref`.
Optional keys are `ipa`, `hook`, `cta`, and `template_id`. Unknown keys, duplicate
keys, multiline values, messages with more than one block, and blocks over 4 KB
are rejected with a validation response. The parser splits only on the first
colon, trims key/value whitespace, normalizes CRLF, and never interprets values
as formulas or code. Before writing to Sheets, prefix values starting with `=`,
`+`, `-`, or `@` with an apostrophe to prevent formula injection.

For every Discord message:

1. Set `source=discord` and keep the submitted dictionary URL in `source_ref`.
   The Discord provenance lives in the immutable ingestion-log `event_id`, so
   editorial verification does not lose its direct reference URL.
2. Compute `event_id=discord:<guild_id>:<channel_id>:<message_id>` and a SHA-256
   `payload_hash` from the canonical parsed fields.
3. Acquire a script lock, then check `ingestion_log.event_id` and
   `content_items.dedupe_key` before appending. A repeated event returns the
   existing `content_id`; a new message with an existing `dedupe_key` is logged
   as `duplicate` and does not create another row.
4. Append valid new rows with `status=draft`,
   `review_status=needs_review`, blank `publish_ok`, and both creation/update
   timestamps. Append the corresponding ingestion log entry before releasing
   the lock.

## Editorial and validation gates

State progression is:

```text
draft -> needs_review -> approved -> claimed -> rendering -> scheduled -> published
```

Failure states are `validation_failed`, `render_failed`, `publish_failed`, and
`reconcile_required`. Retrying updates the same row and reuses `content_id`,
`render_id`, and `publish_id`; it never appends a replacement content item or
creates a second publication.

The editor verifies the sense against `source_ref`, rewrites `meaning_vi` in
plain Vietnamese, checks that both examples express that exact sense and
context, verifies IPA against the chosen dialect, and confirms that hook/CTA do
not make unsupported claims. A second human records their identity in
`reviewed_by`, sets `review_status=approved`, and only then sets
`publish_ok=OK`. Self-review is not accepted.

Automated validation before approval and again before claim requires:

- every required editorial field and a valid HTTPS `source_ref`;
- `language=en`, a controlled part of speech/context/template, and a unique
  `content_id`/`dedupe_key`;
- `example_en` contains the word or an approved inflection and the Vietnamese
  example matches the reviewed sense;
- `review_status=approved`, nonblank `reviewed_by`, `publish_ok=OK`, and
  `status=approved`;
- `publish_at` is blank or a valid future UTC timestamp; operational IDs are
  blank before the first claim.

Any failure sets `status=validation_failed` and a stable `error_code`; it does
not silently repair editorial content.

## Eligible query and atomic claim

The logical eligibility filter is:

```text
publish_ok = "OK"
AND review_status = "approved"
AND status = "approved"
AND reviewed_by != ""
AND (publish_at = "" OR publish_at <= claim_window_end)
AND claimed_at = ""
AND render_id = ""
AND publish_id = ""
```

Google Sheets has no row-level transaction. A single Apps Script function must
hold `LockService.getScriptLock()`, re-read candidate rows, sort by
`publish_at` then `created_at`, and update each selected row in the same locked
section to `status=claimed`, `claimed_at=<now>`, and a deterministic
`render_id=content_id`. It returns only rows whose updates succeeded. A later
run therefore cannot claim them again. If a crash makes downstream state
ambiguous, reconciliation checks the recorded IDs and moves the same row to
`reconcile_required`; it must not clear the IDs and republish blindly.

## Approval boundary

The workbook, Discord credential, Apps Script trigger, scheduler, and public
publisher remain unconfigured/inactive until separate human approval. Testing
uses copied sample rows, manual execution, and a non-public destination only.
